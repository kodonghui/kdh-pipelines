# /star-docs-chain - 문서 파생 체인

소스 문서 1개를 읽고, 선택한 파생 문서를 자동 생성한다.
**소스도 출력도 하드코딩 안 함. 둘 다 사용자에게 되묻는다.**

## 사용법

```
/star-docs-chain                    ← 대화형 (소스+출력 물어봄)
/star-docs-chain meeting            ← 소스=최신 회의, 출력=전부(노션+슬랙+로그)
/star-docs-chain report             ← 소스=최신 보고서, 출력=전부(노션+슬랙)
/star-docs-chain 0409               ← 해당 날짜 문서 자동 탐색
```

인수 없으면 대화형. `meeting`/`report` 단축키로 빠르게 실행 가능.

## 실행 흐름

### Step 1: 소스 확인

**인수가 있으면:**
- `meeting` → `docs/follow-up/` 에서 가장 최근 회의 섹션 자동 탐색
- `report` → `docs/reports/` 에서 가장 최근 `*-report.md` 탐색
- `MMDD` (4자리 숫자) → 해당 날짜가 포함된 문서 탐색 (follow-up + reports 모두)

**인수가 없으면 사용자에게 질문:**
```
소스가 뭐야?
1. 최신 회의록 (팔로업 로그에서 탐색)
2. 최신 보고서 (reports/에서 탐색)
3. 직접 경로 지정
```

**소스 탐색 방법:**

회의록 탐색:
1. `docs/follow-up/` 에서 가장 최근 수정된 `*-follow-up-log.md` 읽기
2. 파일 끝에서 역탐색 → 가장 마지막 `## 20XX-XX-XX | 회의:` 헤더 찾기
3. 그 헤더부터 파일 끝까지 = 소스 텍스트

보고서 탐색:
1. `docs/reports/` 에서 Glob `*-report.md` → 가장 최근 수정일 기준
2. 파일 전체 = 소스 텍스트

**탐색 실패 시:** "못 찾겠습니다. 경로를 직접 알려주세요." (강제 종료 안 함)

**소스를 찾았으면:**
```
소스 확인: [파일명] ([회의록/보고서])
내용 요약: [첫 3줄]
이 소스가 맞나요?
```
사용자 확인 후 다음 Step.

### Step 2: 출력 선택

**단축키로 실행했으면:**
- `meeting` → 노션 TODO + 슬랙 + 팔로업 로그 (3종 전부)
- `report` → 노션 페이지 + 슬랙 (2종)

**대화형이면 사용자에게 질문:**
```
뭘 만들까요? (복수 선택 가능)
1. 노션 페이지
2. 슬랙 메시지
3. 팔로업 로그 엔트리
4. 전부
```

### Step 3: 파생 문서 생성

소스 유형과 선택한 출력에 따라 생성한다. **CLAUDE.md의 형식 규칙을 반드시 따른다.**

#### 3-A: 노션 페이지 생성

**소스가 회의록일 때:**
1. Action Items 추출 → 날짜별 분배 없이 그대로 나열 (날짜별 분배는 사용자 몫)
2. 미결 사항 추출: 상세 내용 중 "추가 논의", "확인 필요", "검토" 등 키워드가 포함된 항목
3. 회의록 원문 복사 (Todo-결론-상세 순서 유지)

출력 구조:
```markdown
# MMDD 회의 후 TODO

## Action Items
- [ ] {항목 1}
- [ ] {항목 2}
...

## 미결 사항
- [ ] {미결 1}
- [ ] {미결 2}

---

# MMDD 회의록 ({시간}, {N}분)

## Todo (Action Items)
{원문 복사}

## 결론
{원문 복사}

## 상세 내용
{원문 복사}

### 일정
{원문 복사}
```

**소스가 보고서일 때:**
CLAUDE.md 노션 형식: **Todo - 결론 - 상세 내용** 순서로 재구성.
1. 보고서에서 "다음 단계" / "향후 작업" 섹션 → Todo
2. 보고서 첫 섹션 또는 "요약" → 결론
3. 나머지 → 상세 내용

출력 구조:
```markdown
# MMDD {보고서 제목}

## Todo
- [ ] {다음 단계 항목들}

## 결론
{요약/핵심 결정사항}

## 상세 내용
{본문 재구성}
```

#### 3-B: 슬랙 메시지 생성

**CLAUDE.md 슬랙 형식 준수:**
```
[주제]
@태그
1. 결론
2. 상세
  a. ...
  b. ...
3. 첨부/일정
```

변환 규칙 (소스 무관):
- **주제**: 소스 제목에서 추출
- **@태그**: 소스에서 참석자/관련자 추출. 못 찾으면 비워두고 사용자에게 물어봄
- **결론**: 소스의 "결론" 또는 핵심 결정사항 (3~5줄로 축약)
- **상세**: 안건/섹션별로 a/b/c 라벨링, 각 3~4개 핵심 불릿 (발화자 인용 제거)
- **일정/첨부**: 소스에 일정 테이블이 있으면 불릿 리스트로 변환

압축 목표: 소스 대비 **70~80% 압축**. 30행 이내.

#### 3-C: 팔로업 로그 엔트리 생성

`docs/follow-up/` 의 기존 팔로업 로그 파일 끝에 append.

엔트리 형식:
```markdown

## YYYY-MM-DD | {유형}: {제목} ({상대방})

### 핵심 결정/변경사항
{불릿 리스트}

### Action Items
| # | 담당 | 항목 | 기한 |
|---|------|------|------|
```

유형 자동 판별:
- 소스에 "회의" 키워드 → `회의`
- 소스에 "보고서" / "report" → `보고`
- 그 외 → `기록`

### Step 4: 미리보기 + 승인 + 저장

1. 생성된 각 문서를 **미리보기로 출력** (전문)
2. 사용자에게 "이대로 저장할까요? 수정할 부분 있으면 말씀해주세요."
3. 승인 시 파일 저장:
   - 노션: `docs/reports/MMDD-notion-{slug}.md`
   - 슬랙: `docs/reports/MMDD-slack-{slug}.md`
   - 팔로업: 기존 파일 끝에 append
4. **INDEX.md 갱신 필요 여부 체크**: 새 파일이 생겼으면 INDEX.md에 반영

### Step 5: GT 미션 연결 제안 (선택적)

생성된 문서(특히 노션 TODO)에서 GT 관련 Action Items를 감지한다.

감지 키워드: "GT 제작", "GT 전달", "negative 제작", "positive", "검수", 지표명(연결성/요청/정합성)

감지되면:
1. 해당 항목에 마커 삽입: `<!-- gt-action: {지표} {한글명} -->`
2. 사용자에게 제안: "/star-gt-mission으로 {지표} 미션을 시작할까요?"
3. 사용자가 거절하면 마커만 남기고 진행

감지 안 되면: 이 Step 스킵. 아무 말 안 함.

## 출력 규칙

- **CLAUDE.md 형식 준수**: 노션(Todo-결론-상세), 슬랙([주제] @태그 형식)
- **em dash 사용 금지**: 구분은 `:`, `-` 사용
- **subcategory 순서**: 요약 - 정보추출 - 추론 (항상 이 순서)
- **승인 없이 파일 저장 금지**: Step 4에서 반드시 미리보기 + 승인

## Quality Self-Check

- [ ] 소스를 사용자에게 확인했는가?
- [ ] 출력 유형을 사용자가 선택했는가?
- [ ] 노션 형식이 CLAUDE.md 규칙(Todo-결론-상세)을 따르는가?
- [ ] 슬랙 형식이 CLAUDE.md 규칙([주제] @태그)을 따르는가?
- [ ] 슬랙 메시지가 30행 이내인가?
- [ ] 발화자 직접 인용이 슬랙에서 제거되었는가?
- [ ] 미리보기를 사용자에게 보여줬는가?
- [ ] 승인 후에만 파일을 저장했는가?
- [ ] INDEX.md 갱신이 필요하면 갱신했는가?

## 유기성

### → star-gt-mission 연결
Step 5에서 GT 관련 Action Items 감지 시 마커 삽입 + /star-gt-mission 시작 제안.
독립 실행도 가능 - 연결은 선택적.

### ← kdh-analyze/kdh-plan 연결
이 스킬은 분석/계획 체인과 독립. `_bmad-output/kdh-plans/`에 저장하지 않음.
문서 파생은 분석 작업이 아니라 산출물 작업이므로 별개 계열.
