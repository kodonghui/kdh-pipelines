# /kdh-analyze — CEO 전용 깊은 분석 명령어

사장님이 매번 "어떻게 할지 메타적으로 상세히 분석해서 구체적이고 자세하게 아이디어와 방법을 나한테 설명해줘 Ultrathink" 를 안 쳐도 되게 만든 명령어.

## 사용법

```
/kdh-analyze [주제 또는 질문]
```

## 실행 시 자동 적용되는 것

이 명령어를 받으면 아래 규칙을 자동으로 적용한다:

### 1. Extended Thinking (최대)
- 최대 사고 토큰 사용 (31,999~63,999)
- plan 모드 대신 extended thinking으로 깊은 분석 수행
- plan 모드는 구조화에는 좋지만, 자유로운 메타 분석에는 extended thinking이 더 효과적

### 2. 분석 프레임워크 (자동 적용)

주어진 주제에 대해 아래 순서로 분석한다:

**Phase 1: 메타 분석 (왜?)**
- 이 주제가 왜 중요한지
- 현재 상태는 어떤지
- 어떤 문제를 풀려는 건지

**Phase 2: 다방면 분석 (뭐가 있지?)**
- 가능한 접근법 3~5가지 나열
- 각각의 장단점
- 업계 사례/벤치마크 (있으면)
- 우리 프로젝트에 맞는지

**Phase 3: 아이디어 (이렇게 하면?)**
- 구체적 아이디어 제시
- 기존 방식과 뭐가 다른지
- 예상 효과

**Phase 4: 실행 방안 (어떻게?)**
- 구체적 구현 방법 (파일, 코드, 명령어)
- 순서와 의존관계
- 예상 시간/난이도
- 사장님이 결정해야 할 선택지

### 3. 출력 규칙

- **한국어** — 기술 용어 최소화, 비유로 설명
- **구체적** — "좋은 방법"이 아니라 "이 파일의 이 부분을 이렇게"
- **선택지** — 마지막에 항상 A/B/C 선택지 제시
- **짧은 요약 먼저** — TL;DR 3줄 → 상세 분석 순서

### 4. 리서치 포함 (필요 시)

주제가 외부 정보를 필요로 하면 자동으로:
- WebSearch로 최신 정보 검색
- GitHub에서 실제 구현 사례 검색
- 결과를 분석에 통합

리서치가 필요 없는 내부 주제면 바로 분석.

## 예시

```
사장님: /kdh-analyze 에이전트 실행 엔진을 Phase 2에서 어떻게 만들지

→ 자동으로:
  1. Phase 2 에이전트 엔진의 의미/중요성 분석
  2. 가능한 접근법 비교 (Claude SDK 직접, Agent SDK, Hermes 등)
  3. 구체적 아이디어와 아키텍처 제안
  4. 파일 구조, API 설계, 구현 순서 제시
  5. A/B/C 선택지
```

```
사장님: /kdh-analyze 왜 Sprint 0가 두 번 삭제됐는지 원인분석

→ 자동으로:
  1. 삭제 이력과 원인 추적
  2. 근본 원인 분류 (프로세스/기술/소통)
  3. 재발 방지 아이디어
  4. 구체적 개선 방안과 적용 시점
  5. 사장님 결정 필요 사항
```

## /kdh-analyze 실행 (Execution Mode)

분석 결과를 사장님이 승인하면, 같은 세션에서 바로 실행 가능:

```
사장님: /kdh-analyze [주제]
→ 분석 + 선택지 제시

사장님: B로 해
→ B안 기반으로 즉시 구현 시작 (extended thinking 유지)
```

## plan 모드 대신 이걸 쓰는 이유

| plan 모드 | /kdh-analyze |
|----------|-------------|
| 구조화된 단계별 계획 | 자유로운 메타 분석 + 아이디어 |
| 승인 후 실행 (2단계) | 분석→승인→실행 (한 흐름) |
| 코드 변경 중심 | 전략/방향/설계 중심 |
| 항상 파일 수정 제안 | 필요하면 리서치, 필요하면 구현 |

둘 다 쓸 수 있음. 복잡한 코드 변경 = plan 모드, 전략적 분석 = /kdh-analyze.
