# /star-workflow - 오늘의 워크플로우

오늘 할 과업들의 **실행 순서**를 설계하고, .md 파일로 저장한다.
터미널에는 요약만 보여주고, 파일을 열어서 확인하도록 안내한다.

## 사용법

```
/star-workflow              <- 오늘 워크플로우 새로 생성
/star-workflow update       <- 기존 워크플로우 수정
/star-workflow status       <- 현재 진행 상태 확인
/star-workflow MMDD         <- 해당 날짜 워크플로우 조회
```

인수 없으면 오늘 워크플로우 생성. `update`면 오늘 기존 워크플로우 수정.

## star-help와의 경계

| | star-help | star-workflow |
|---|---|---|
| 질문 | "뭐 해야 하지?" | "어떤 **순서**로 하지?" |
| 출력 | 현황 + A/B/C 선택지 | 순서 워크플로우 + 순서도 |
| 수정 | 없음 (읽기 전용) | 사용자 수정 가능 |
| 저장 | 안 함 | docs/workflow/ 에 저장 |
| 학습 | 안 함 | 패턴 누적 |

star-help = 재료 수집, star-workflow = 레시피 작성.

---

## 핵심 규칙

### 시간표 금지
- **"몇시에 뭐하고" 식의 시간 배분을 절대 하지 않는다**
- HH:MM, 블록(09:30~11:00), 예상 N분 등 시간 관련 표현 금지
- 순서와 흐름만 표현한다

### 파일 먼저, 터미널은 요약만
- mermaid, 상세 내용은 .md 파일 안에만 존재
- 터미널에 mermaid raw 코드를 절대 출력하지 않는다
- 터미널 출력은 5줄 이내: 파일 경로 + 과업 번호/제목 목록

## 실행 흐름

### Phase 1: 소스 스캔 + 맥락 읽기

**읽기만 한다. 파일 수정 안 함.**

#### 1-A: 상태 스캔 (star-help 동일)

```
1. CLAUDE.md "현재 GT 작업 현황" 테이블 읽기
2. docs/reports/ 최신 notion-todo 읽기
   - 체크박스 파싱: 미완료([ ]) 항목 추출
   - 미결 사항 추출
   - 일정 테이블에서 데드라인 추출
3. docs/follow-up/ 최신 팔로업 로그 읽기
   - 마지막 섹션의 Action Items (미완료)
   - 외부 의존 항목 ("대기", "수신", "전달")
4. gt/ 미션 폴더 스캔
   - 진행중 미션의 잔여 작업량
5. 패턴 메모리 참조 (있으면)
   - docs/workflow/_patterns.md 읽기
```

star-help 결과가 같은 세션에 있으면 1-A 재스캔 생략 가능.

#### 1-B: 과업 맥락 깊이 읽기 (중요)

동희님가 과업을 지정하면, 해당 과업과 관련된 자료를 깊이 읽는다.

```
1. 동희님 지정 과업 파악
2. 관련 자료 탐색:
   - docs/follow-up/ 회의록에서 해당 과업 언급된 섹션
   - docs/reports/ 보고서에서 관련 내용
   - gt/ 해당 지표 폴더의 프롬프트, 결과, 로그
   - docs/research/ 관련 리서치 자료
3. 각 과업에 "참고 자료" 목록 작성 (파일 경로)

예시: 동희님가 "frame 재정의" 지시
  -> 0407 회의록 안건1 (도영님 "추론이면 오케이" 발언)
  -> gt/0406-request/prompts/ 기존 frame 프롬프트
  -> gt/0406-request/results/fail-gt/frame/ 기존 결과물
  -> gt/0406-request/log/ 작업 로그
```

### Phase 2: 순서 결정

수집된 과업을 **우선순위 로직**에 따라 정렬한다.

#### 우선순위 (높은 순)

```
1. 데드라인 지연 (과거 날짜 미완료) -> 즉시
2. 데드라인 임박 (D-2 이내) -> 관련 작업 우선
3. 외부 의존 확인 (대기 항목 체크)
   - 해소됨 -> 해당 작업 시작
   - 미해소 -> 대기 표시, 대안 작업으로 전환
4. 진행중 GT 잔여분 -> 이어서 작업
5. 미결 사항 (대기 시간 활용)
6. 회의 D-1 -> 준비 문서
7. 새 미션 시작
```

동희님가 순서를 직접 지정하면 우선순위 로직보다 동희님 지시를 우선한다.

#### 분기 처리

외부 의존이 있으면 **조건 분기**로 표현:
```
"재형님 Positive GT 수신?"
├── Yes -> A 경로 (검수 + fail GT)
└── No  -> B 경로 (미결 사항 처리)
```

### Phase 3: 파일 저장 + 터미널 요약

**파일을 먼저 만들고, 터미널에는 요약만 보여준다.**

#### 3-A: .md 파일 생성

```
파일: docs/workflow/MMDD-workflow.md
```

파일 구조:

```markdown
# 워크플로우: YYYY-MM-DD (요일)

> 생성: /star-workflow | 상태: 확인 대기

## 상황

{오늘의 상황 요약 1-2줄}

## 순서

1. {과업 제목}
   - 상세: {1줄 설명}
   - 산출물: {뭐가 나와야 하는지}
   - 참고: {관련 파일 경로 목록}

2. {과업 제목}
   - 상세: {1줄 설명}
   - 산출물: {뭐가 나와야 하는지}
   - 참고: {관련 파일 경로 목록}

--- 조건 분기 ---
> "{질문}?"
> - Yes -> {과업 A}
> - No -> {과업 B}

3. {과업 제목}
   ...

## 데드라인

- (!) {날짜} {내용}

## 메타

- 총 과업: {N}개
- 데드라인: {있으면 표시}
- 패턴 참조: {_patterns.md에서 참조한 패턴, 없으면 "없음"}
```

#### 3-B: 순서도 (mermaid in .md)

.md 파일 안에 mermaid 코드블록을 포함한다.
VS Code에서 Ctrl+Shift+V (마크다운 프리뷰)로 순서도가 렌더링된다.
(필수 확장: bierner.markdown-mermaid)

#### mermaid 순서도 규칙

- 노드 최대 **10개** (과업이 많으면 늘려도 됨)
- 분기는 **2단계까지** (깊으면 텍스트로 보충)
- 색상: 완료=초록, 진행중=파랑, 대기=회색, 긴급=빨강
- classDef로 색상 정의

**노드 텍스트 규칙 (중요):**
- 제목만 달랑 쓰지 말 것. **구체적으로 뭘 하는지** 노드 안에 적을 것
- `<br/>`로 줄바꿈하여 2-3줄로 상세하게
- 좋은 예: `["1. 프레임 재정의<br/>구조 constraint 412건 유형별<br/>판정 기준 + 분류 필요 91건 개별 판정"]`
- 나쁜 예: `["1. 프레임 재정의"]`
- 산출물이 있으면 노드에 포함: `<br/>-> 0408-frame-review.md`
- 데이터 수치가 있으면 노드에 포함: `<br/>412건 분석 + 91건 분류`

#### 3-C: 터미널 출력 (5줄 이내)

```
워크플로우 저장:
  docs/workflow/MMDD-workflow.md
  docs/workflow/MMDD-workflow-flowchart.html  <- 브라우저에서 열기

1. {과업 제목}
2. {과업 제목}
3. {과업 제목}
```

**터미널에 mermaid raw 코드, 상세 설명, 참고 자료 등을 출력하지 않는다.**

### Phase 4: 동희님 확인

```
파일 열어서 확인해주세요. 수정할 거 있으면 말씀해주세요.
"좋아" 하면 확정합니다.
```

**수정 요청 처리:**
- "2번이랑 3번 바꿔" -> .md 파일 수정 + 순서도 재생성
- "이것도 추가해" -> .md 파일에 과업 추가
- "이건 빼" -> .md 파일에서 과업 제거

수정 후 파일 갱신 + 터미널에 변경 요약만 출력.

승인 후: 파일 상단 상태를 "확인 대기" -> "동희님 승인"으로 변경.

### Phase 5: 패턴 기록 (ECC 연동용)

워크플로우 승인 후, 패턴 학습용 데이터를 기록한다.

```
파일: docs/workflow/_patterns.md (없으면 자동 생성, 있으면 append)
```

기록 내용:

```markdown
## YYYY-MM-DD (요일)

계획 순서: [1, 2, 3, 4, 5]
실제 순서: [미기록 - 세션 종료 시 /star-workflow status로 확인]
동희님 수정: [있었으면 뭘 바꿨는지]
외부 의존: [있었으면 해소 여부]
```

**패턴 활용 시점:**
- _patterns.md에 5일치 이상 쌓이면 패턴 추출 시작
- 추출 대상:
  - "외부 대기 있으면 {Z}부터 하는 경향" (상황별 패턴)
  - "동희님 수정이 자주 발생하는 부분" (개선 포인트)
- ECC-3h: docs/workflow/ 스캔, 계획 vs 실제 비교
- ECC-12h: _patterns.md에서 반복 패턴 추출, 다음 워크플로우에 반영

## /star-workflow update

기존 오늘 워크플로우를 수정한다.

```
1. docs/workflow/MMDD-workflow.md 읽기
   - 없으면: "오늘 워크플로우가 없습니다. /star-workflow로 먼저 만드세요."
2. 현재 상태 표시:
   - 각 과업 옆에 [완료/진행중/대기] 표시
3. 사용자 수정 반영:
   - 순서 변경, 추가, 삭제
4. mermaid 재생성
5. 파일 덮어쓰기 (기존 파일 갱신)
6. _patterns.md에 수정 내역 기록
```

## /star-workflow status

현재 워크플로우 진행 상태를 확인한다.

```
1. docs/workflow/MMDD-workflow.md 읽기
2. 실제 진행 상태 추정:
   - gt/ 폴더 결과물 카운트 변화
   - docs/ 새 파일 여부
   - 팔로업 로그 마지막 엔트리
3. 터미널 출력:

(v) 1. 재형님 슬랙 확인
(>) 2. 프레임 지표 재검토
( ) 3. 휴먼 검수 평가 지표
( ) 4. 정리 + save-session

진행률: 1/4 (25%)
```

## 공통 규칙

### CLAUDE.md 준수
- subcategory 순서: 요약 - 정보추출 - 추론
- em dash 사용 금지
- 파일명 MMDD prefix + kebab-case
- **시간표 금지**: 워크플로우에 시간 배분하지 않음

### 읽기/쓰기 분리
- Phase 1~2: 읽기만
- Phase 3: 파일 저장 (터미널은 요약만)
- Phase 4: 동희님 확인 대기
- Phase 5: 승인 후 패턴 기록

### 한국어 출력
- 모든 출력은 한국어
- mermaid 노드 텍스트도 한국어

## 유기성

### <- star-help 연결
Phase 1 소스 스캔에서 star-help와 동일한 데이터 소스 사용.
star-help 결과가 같은 세션에 있으면 재스캔 생략 가능.

### -> star-gt-mission 연결
워크플로우에 "새 미션 셋업" 과업이 있으면:
"이 단계에서 /star-gt-mission 실행할까요?" 제안.

### -> star-docs-chain 연결
워크플로우에 "문서 생성" 과업이 있으면:
"이 단계에서 /star-docs-chain 실행할까요?" 제안.

### <-> ECC 연동
- ECC-3h: docs/workflow/ 스캔 -> 계획 vs 실제 비교 -> 차이 기록
- ECC-12h: _patterns.md 패턴 추출 -> 다음 워크플로우 개선
- memory: 확신도 높은 패턴 -> memory/feedback_workflow_patterns.md 저장

### <-> save-session / resume-session
- save-session: 워크플로우 진행 상태도 세션에 포함
- resume-session: 워크플로우 잔여 과업 표시
