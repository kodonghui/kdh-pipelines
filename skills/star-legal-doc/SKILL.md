# /star-legal-doc - 법학 논문 형식 문서 생성

소스(분석 결과, 논의 로그, 자유 주제)를 읽고, 법학 논문 형식의 구조화된 문서를 생성한다.

## 사용법

```
/star-legal-doc                          <- 대화형 (소스 물어봄)
/star-legal-doc path/to/source.md        <- 소스 지정
/star-legal-doc 주제를 자유 기술          <- 현재 세션 맥락 기반
```

## 문서 구조

### 상단: CLAUDE.md 노션 형식 (필수)

```markdown
# MMDD {제목}

## Todo
- [ ] [담당자] {Action Item}
...

## 결론
{핵심 결정사항/제안 요약. 3~5줄.}

---

## 상세 내용
```

**순서: Todo - 결론 - 상세 내용.** 이 순서는 CLAUDE.md 규칙이므로 변경 불가.

### 상세 내용: 법학 논문 형식 (I ~ VIII)

상세 내용 안에서 아래 8장 구조를 따른다.
주제에 따라 장을 생략하거나 합칠 수 있지만, 번호 체계(I, II, III...)는 유지한다.

#### I. 문제의 제기

- 현재 상태와 문제를 기술
- "왜 이 문서를 쓰게 되었는가?"에 답하는 장
- 문제 N개를 **문제 1**, **문제 2** 형식으로 나열
- 각 문제는 현상 -> 왜 문제인가 -> 영향 순서로 서술
- 5건 사례 나열이 아니라 "성격이 다른 조건들이 섞여있다" 식으로 본질을 짚을 것

#### II. 배경/근거

- 이 문제를 해결하기 위해 조사/분석한 내용
- 학술 근거, 데이터 분석, 기존 논의 결과 등
- 표로 정리 가능하면 표 사용
- 소스에 리서치 결과가 있으면 핵심만 발췌

#### III. 제안

- 해결 방안 제시
- 안이 여러 개면 **A안**, **B안** 형식으로 병렬 제시
- 각 안: 한 줄 요약 + 구조/상세 + 사례
- 비교표가 있으면 이 장에 포함

#### IV. 각 안의 조작적 정의와 사례

- III에서 제시한 안의 구체적 정의
- **실제 데이터 사례를 반드시 포함**: query, requirements, constraint, answer 상태별 판정
- 사례는 표로 정리: 상황 | 판정A | 판정B 형식
- 사례 없이 추상적 정의만 쓰지 말 것

#### V. 제외/예외 기준

- 적용하지 않는 것, 범위 밖인 것을 명시
- 근거와 함께 기술 (누가 결정했는지, 왜 제외하는지)

#### VI. 기존 산출물 영향

- 이미 만든 GT, 보고서, 데이터에 미치는 영향
- 변경/유지/삭제 대상을 표로 정리

#### VII. 비교/평가

- 안이 여러 개일 때: 기준별 비교표
- 비교 기준은 주제에 맞게 선정 (예: 진단 해상도, 운영 비용, 외부 설명 용이성)
- 안이 1개면 이 장 생략 가능

#### VIII. 향후 일정/다음 단계

- 확인 요청 사항 (도영님/재형님 등에게 물어볼 것)
- 단계별 실행 순서
- 선행 조건 명시

## 실행 흐름

### Step 1: 소스 확인

인수가 있으면:
- 파일 경로 -> 해당 파일 읽기
- 자유 텍스트 -> 현재 세션 맥락 + 프로젝트 파일에서 관련 내용 수집

인수가 없으면:
```
소스가 뭐야?
1. 분석 결과 (analysis/ 탐색)
2. 논의 로그 (log/ 탐색)
3. 직접 경로 지정
4. 현재 세션 맥락 사용
```

### Step 2: 구조 설계

소스를 읽고 8장 중 어떤 장이 필요한지 판단.

- 안이 1개: VII(비교) 생략
- 안이 2개 이상: VII 필수
- 기존 산출물 없음: VI 생략
- 조사/학술 근거 없음: II를 "배경"으로 축소

사용자에게 확인:
```
구조:
I. 문제의 제기 - {한 줄}
II. 배경/근거 - {한 줄}
III. 제안 - A안 vs B안
IV. 조작적 정의와 사례
V. 제외 기준
VI. 기존 GT 영향
VII. 비교
VIII. 다음 단계

이 구조로 갈까요?
```

### Step 3: 문서 생성

8장 구조에 따라 문서 작성. 다음 규칙 준수:
- em dash 사용 금지 (`:`, `-` 사용)
- subcategory 순서: 요약 - 정보추출 - 추론
- 워크플로우에 시간 배분 금지 (순서만)
- 사례는 실제 데이터 기반 (ID, query, answer 포함)
- 표 적극 활용

### Step 4: 미리보기 + 승인

1. 전문 미리보기 출력
2. "이대로 저장할까요? 수정할 부분 있으면 말씀해주세요."
3. 승인 시 `docs/reports/MMDD-notion-{slug}.md`에 저장
4. INDEX.md 갱신

### Step 5: 슬랙 메시지 제안 (선택)

노션 문서 생성 후:
```
슬랙 메시지도 만들까요?
1. 팀 공유용
2. 확인 요청용 (도영님/재형님)
3. 둘 다
4. 안 만듦
```

선택 시 /star-docs-chain의 슬랙 형식([주제] @태그)으로 생성.

## 작성 톤

- 학술 논문이 아니라 **실무 보고서**
- 격식체 (합쇼체) 사용
- 근거 있는 주장만 (추측은 "추정" 명시)
- 읽는 사람(도영님/재형님)이 **5분 안에 핵심을 파악**할 수 있어야 함
- Todo와 결론만 읽어도 의사결정이 가능해야 함
