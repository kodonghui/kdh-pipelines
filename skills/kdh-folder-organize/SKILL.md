# /kdh-folder-organize - 폴더 정리 파이프라인

프로젝트 폴더를 안전하게 정리하는 5단계 파이프라인.
**삭제 절대 금지. 항상 아카이브 우선.**

## 사용법

```
/kdh-folder-organize              ← 기본 = scan
/kdh-folder-organize scan         ← Phase 1: 현황 분석 + 문제점 감지
/kdh-folder-organize plan         ← Phase 2: 정리 계획 제시 (승인 대기)
/kdh-folder-organize run          ← Phase 3+4: archive + 이동 실행
/kdh-folder-organize index        ← Phase 5: INDEX.md 생성/갱신
/kdh-folder-organize full         ← Phase 1~5 순차 실행 (매 Phase 승인)
```

인수 없으면 `scan` 실행. `full`은 각 Phase 끝마다 사용자 승인을 받고 다음 진행.

## 4명령어 워크플로우 참고

이 스킬은 독립 실행 도구이며, discuss/research/analyze/plan 체인에 직접 연결되지 않음.
단, 대규모 폴더 재구성이 필요하면 먼저 `/kdh-discuss 폴더 정리 방향`으로 논의 후 실행 권장.

## 공통 규칙 (모든 Phase에 적용)

### 안전 원칙
1. **Archive-first**: 파일 삭제 절대 금지. 정리 = `_archive/`로 이동
2. **Dry-run default**: `run` Phase 전에 반드시 `plan`으로 미리보기
3. **검증 필수**: 이동 후 파일 수 비교. 원본과 결과 불일치 시 중단

### 구조 원칙
4. **Category > Date**: 최상위 = 주제/도메인/지표별, 하위 = MMDD 날짜별
5. **3-4 level max**: 루트 포함 4레벨까지. 5레벨 이상은 ERROR
6. **MMDD prefix**: 파일명에 날짜 접두사 (예: `0408-research.md`)
7. **kebab-case**: 폴더/파일명은 소문자+하이픈. 한국어 폴더명 허용

### 제외 대상 (스캔/이동 제외)
- `.claude/`, `.git/`, `node_modules/`, `.next/`, `__pycache__/`
- `.gitignore`, `package-lock.json`, `*.lock`

## Phase 1: SCAN

현재 프로젝트의 폴더/파일 현황을 분석하고 문제점을 감지한다.

### 실행 절차

1. **프로젝트 루트 확인**: 현재 working directory 기준
2. **전체 파일 트리 스캔**:
   ```bash
   find . -type f -not -path '*/.claude/*' -not -path '*/node_modules/*' \
     -not -path '*/.git/*' -not -path '*/.next/*' -not -path '*/__pycache__/*' | sort
   ```
3. **통계 집계**:
   - 총 파일 수, 총 폴더 수
   - 확장자별 파일 수 (상위 10개)
   - 폴더별 파일 수 (상위 10개)
   - 최대 깊이

4. **문제점 감지 (8종)**:

| # | 항목 | 감지 방법 | 심각도 |
|---|------|----------|--------|
| 1 | 빈 폴더 | `find -type d -empty` (제외 대상 제외) | WARN |
| 2 | 임시 파일 | `_temp*`, `*.tmp`, `nul`, `_build_*` 패턴 | WARN |
| 3 | 깊이 5+ | `find -mindepth 5 -type d` | ERROR |
| 4 | 날짜 없는 파일명 | MMDD 패턴 미매칭 (.md, .json 대상) | INFO |
| 5 | 중복 파일명 | 같은 이름이 다른 폴더에 존재 | WARN |
| 6 | 빌드 아티팩트 | node_modules, .next, dist, build 등 | INFO |
| 7 | 루트 오염 | 루트에 코드/데이터 파일 (CLAUDE.md, README 제외) | WARN |
| 8 | 대용량 파일 | 50MB 이상 | INFO |

5. **출력 포맷**:

```
## SCAN 결과: [프로젝트명]
> 스캔: YYYY-MM-DD | 파일: N개 | 폴더: N개 | 최대 깊이: N

### 통계
| 확장자 | 파일 수 |    | 폴더 | 파일 수 |
|--------|--------|    |------|--------|

### 문제점
| # | 심각도 | 항목 | 대상 | 설명 |
|---|--------|------|------|------|
ERROR 먼저, WARN 다음, INFO 마지막.

### 폴더 트리
(find 결과 기반 트리 구조)
```

6. **사용자에게 결과 보여주고 대기**: "문제점 N개 발견. `/kdh-folder-organize plan`으로 정리 계획을 세울까요?"

## Phase 2: PLAN

SCAN 결과를 기반으로 정리 계획을 세운다. **실제 파일 이동 없음.**

### 실행 절차

1. **프로젝트 타입 자동 감지**:
   - JSONL/데이터 파일 다수 → 데이터 프로젝트
   - src/, package.json → 코드 프로젝트  
   - docs/ 위주 → 문서 프로젝트
   - 감지 결과를 사용자에게 확인: "데이터+문서 프로젝트로 판단. 맞나요?"

2. **목표 구조 설계**:
   - Category > Date 원칙 적용
   - 프로젝트 타입에 맞는 최상위 폴더 제안
   - 기존 파일을 새 구조에 매핑

3. **이동 계획 테이블 출력**:

```
## 정리 계획

### 목표 구조
(새 폴더 트리)

### 이동 계획
| # | 현재 위치 | 이동 위치 | 이유 |
|---|----------|----------|------|

### 삭제 대상 (= archive 이동)
| # | 파일 | 이유 |
|---|------|------|

### 변경 없음 (그대로 유지)
| # | 파일/폴더 | 이유 |
|---|----------|------|
```

4. **사용자 승인 대기**: "이 계획대로 진행할까요? 수정할 부분이 있으면 말씀해주세요."
   - 승인 시 → Phase 3으로
   - 수정 요청 시 → 계획 수정 후 다시 제시
   - **승인 없이 절대 다음 Phase 진행 금지**

## Phase 3+4: ARCHIVE + MOVE (run)

사용자가 `plan`을 승인한 후에만 실행. **plan 없이 run 직접 호출 시 거부.**

### Phase 3: ARCHIVE

1. **디스크 용량 체크**:
   ```bash
   df -h .
   ```
   남은 용량이 프로젝트 크기의 2배 미만이면 경고.

2. **아카이브 폴더 생성**:
   ```bash
   mkdir -p _archive/YYYYMMDD-before-reorg
   ```

3. **이동 대상 파일만 아카이브에 복사** (전체 복사 아님):
   - plan에서 "이동" 또는 "삭제(archive)" 표시된 파일만 복사
   - `cp -r` 사용 (mv 아님)
   
4. **복사 검증**: 복사된 파일 수 = plan 대상 파일 수

### Phase 4: MOVE

1. **새 폴더 구조 생성**: `mkdir -p` 로 목표 폴더들 생성
2. **파일 이동**: plan 테이블 순서대로 `mv`
3. **이동 검증**:
   - 이동 전 파일 수 == 이동 후 파일 수 (제외 대상 빼고)
   - 빈 폴더 정리: 이동 후 비어진 원래 폴더 → `_archive/`로 이동
4. **결과 보고**:
   ```
   ## 이동 완료
   - 이동: N개 파일
   - 아카이브: N개 파일  
   - 새 폴더: N개 생성
   - 검증: OK / FAIL
   ```

5. FAIL 시 **즉시 중단** + 롤백 안내:
   ```
   _archive/YYYYMMDD-before-reorg/ 에서 원본 복구 가능
   ```

## Phase 5: INDEX

프로젝트 루트에 INDEX.md를 생성하거나 갱신한다.

### 실행 절차

1. **전체 트리 재스캔** (이동 후 최신 상태)
2. **INDEX.md 생성**:

```markdown
# [프로젝트명] - File Index
> Last updated: YYYY-MM-DD | Files: N | Folders: N
> Generated by: /kdh-folder-organize

## Active (현재 작업)
| 폴더 | 설명 | 파일 수 | 상태 |
|------|------|--------|------|

## Archive (아카이브)
| 폴더 | 설명 | 파일 수 | 아카이브 날짜 |
|------|------|--------|-------------|

## Folder Tree
```
(tree 출력)
```

## 파일 네이밍 규칙
- MMDD prefix + kebab-case
- 예: `0408-research-report.md`
```

3. **CLAUDE.md 경로 갱신 제안**:
   - CLAUDE.md 읽기
   - 변경된 경로 목록 출력
   - "CLAUDE.md에서 이 경로들을 갱신할까요?" 질문
   - **자동 수정 아님 - 사용자 승인 후 수정**

4. **완료 보고**:
   ```
   INDEX.md 생성 완료.
   CLAUDE.md 갱신 제안: N건
   ```

## full 모드

Phase 1~5를 순차 실행하되, 각 Phase 완료 후 사용자 승인을 받는다.

```
[Phase 1: SCAN] → 결과 출력 → "계속?" 
→ [Phase 2: PLAN] → 계획 출력 → "승인?"
→ [Phase 3+4: ARCHIVE+MOVE] → 결과 출력 → "INDEX 생성?"
→ [Phase 5: INDEX] → 완료
```

어느 단계에서든 사용자가 "중단"하면 즉시 멈춘다.
