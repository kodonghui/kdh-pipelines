# Plan: KDH 시리즈 v11.1 — "눌러봤냐?" + 멀티모델 MoA 통합

## Context

### 왜 이걸 하는가?

Phase 1을 D1-D6 리뷰(3 critic, cross-talk, auto-fail)로 밤새 돌려서 Sprint 1~5 전부 PASS(7.63~7.93) 받았는데, **CEO가 브라우저에서 열어보니 기본적인 것들이 안 됨:**

| 버그 | 심각도 | 파이프라인이 잡았나? |
|------|:-----:|:------------------:|
| Content-Length: 0 — 사이트 자체가 안 열림 | P0 | ❌ |
| "세션 만료" — 처음 방문인데 만료 메시지 | P1 | ❌ |
| 회원가입 클릭 → 즉시 로그인으로 튕김 | P1 | ❌ |
| 초대 링크 → localhost:5173 (접속 불가) | P1 | ❌ |
| Subframe 디자인 미적용 — 기획과 다른 UI | P2 | ❌ |

**근본 원인: 코드 리뷰는 코드 품질을 봤지, "회원가입 버튼 누르면 되나?"를 아무도 안 눌러봤음.**

### v10.1과의 관계

v10.1(이전 계획)의 D1-D6 강제, cross-talk, auto-fail 개선은 **그대로 포함**하되, v11에서 추가하는 핵심은:

1. **브라우저 스모크 테스트 = BLOCKING** (코드 리뷰보다 먼저)
2. **평균 escape hatch 제거** (D6 낮아도 평균으로 PASS 금지)
3. **Subframe 디자인 검증 GATE** (디자인 안 쓰면 빌드 거부)
4. **하드코딩 정적 분석** (localhost URL 자동 탐지)
5. **배포 후 프로덕션 검증** (인프라 포함)

## 근본 문제 4가지

### 1. "코드 품질 ≠ 제품 정확성"

```
현재: kdh-build → kdh-review(코드 리뷰) → PASS → 다음 스토리
문제: 3명이 코드를 읽었지만 브라우저를 안 열었음
```

**수정:** 스토리 빌드 후, 코드 리뷰 전에 **브라우저 스모크 체크** 삽입

### 2. "D6 평균 escape hatch"

```
현재: Quinn(D6=6.0) + Winston(8.5) + John(8.5) = 평균 7.67 → PASS
결과: E2E 테스트 0개인 코드가 PASS
```

**수정:** 모든 D ≥ 7.0 필수. 평균 아닌 개별 minimum threshold.

### 3. "디자인 → 개발 연결 끊김"

```
현재: kdh-plan Stage 5에서 Subframe 디자인 → kdh-build에서 직접 React 코딩
결과: Subframe 디자인 무시, 자체 컴포넌트
```

**수정:** UI 스토리는 Subframe 컴포넌트 import 검증 GATE 추가

### 4. "인프라 blindspot"

```
현재: 파이프라인이 localhost에서만 테스트. nginx, Cloudflare, 프로덕션 URL 테스트 0
결과: nginx 버그로 사이트 전체 불통
```

**수정:** 배포 후 프로덕션 URL로 Playwright 검증 필수

```
       □ JS 에러 없음

     - 렌더링 OK + 에러 없음 → PASS → Phase 5로 진행

     - 3회 FAIL → ESCALATE

  추가: Full-stack 스토리는 user flow 테스트:
     - 예: signup 스토리 → 가입 폼 채우기 → 제출 → 성공 화면 확인
     - Playwright 자동화 또는 수동 스크린샷
```

```
Phase 0.5: Infrastructure Check (신규, BLOCKING)

     - curl -sD - http://127.0.0.1:80/ -H "Host: corthex-hq.com" | head -20

     - curl -sI https://corthex-hq.com/ → HTTP 200?

     - JS 파일 tail에 "Content-Length: 0" 없는지 확인

  3. SSL 유효성:
     - curl --cert-status https://corthex-hq.com/ → 유효?
     
  INFRA-FAIL → 수정 후 재실행 (E2E 진행 불가)
```

## 검증 방법

### v11 신규 검증 (v10.1 검증은 그대로 유지)

**1. "눌러봤냐?" 검증**

## 8. OpenAI MCP + 멀티모델 통합 (v11.1 신규)

### 왜 이걸 하는가?

Claude Code에 OpenAI MCP가 들어왔고, GPT가 Claude보다 잘하는 영역이 있음:
- **보안 스캔**: GPT가 injection/XSS 패턴 탐지에 더 강함
- **어드버셜 리뷰**: 설계 결정에 반론을 제기하는 데 GPT가 더 공격적
- **백그라운드 위임**: Claude가 빌드하는 동안 GPT가 별도 조사
- **비용**: o3가 Opus보다 3배 저렴한 출력 비용

### 실제 사례 (인스타 @charlieautomates)

```
Use Case 1: /codexreview
  → Claude가 만든 코드를 GPT가 리뷰 (보안 구멍, 엣지 케이스, 설계 결함)
  → "두 번째 시니어 개발자가 30초 만에 리뷰"

Use Case 2: /codex:adversarial-review
  → 설계 결정을 압박 테스트. 가정을 도전. 실패 모드 발견.
  → "두 번째 AI 두뇌로 스트레스 테스트"

Use Case 3: /codex:rescue "버그 조사해줘"
  → Claude가 빌드 계속하는 동안 GPT가 백그라운드에서 조사
  → 끝나면 /codex:result로 결과 가져오기
```

### KDH 파이프라인에 적용

#### 적용 1: kdh-review에 GPT 4번째 크리틱 추가

```
현재: winston(Opus) + quinn(Opus) + john(Opus) → 3명 리뷰
v11.1: winston + quinn + john + codex(GPT) → 4명 리뷰

codex 크리틱 역할:
  - 보안 전문: injection, XSS, CSRF, 인증 우회
  - 어드버셜: "이 설계가 왜 안 되는지" 반론 제기
  - 가중치: D6(리스크인식) 30%, D2(완전성) 25%, 나머지 균등

구현: OpenAI MCP 호출
  → mcp__openai__chat({ model: "o3", prompt: "Review this code for security..." })
```

#### 적용 2: kdh-build에 백그라운드 GPT 조사

```
Phase 2(Implementation) 중 어려운 버그 발생 시:
  → GPT에 백그라운드 위임: "이 에러 왜 나는지 조사해"
  → Claude는 다른 부분 계속 구현
  → GPT 결과 돌아오면 반영
```

#### 적용 3: kdh-code-review-full-auto Phase 4를 MoA로

```

  보안 크리틱:
    Reference: [Opus, GPT-5.4, o3]  → GPT가 injection 잡고, o3가 공격 체인 추론
    Aggregator: Opus → 최종 판정

    Reference: [Opus, Gemini-3-Pro, GPT-5.4]  → Gemini가 1M 컨텍스트로 전체 코드 분석

    Reference: [Opus, Gemini-Vision, GPT-5.4]  → Gemini가 스크린샷 vs 디자인 비교

```

```
  → ~/.claude/mcp.json에 openai 서버 추가
  → OPENAI_API_KEY 설정
  → 테스트: mcp__openai__chat 호출

  → 현재 코드 변경분(git diff)을 GPT에 보내서 리뷰 받기
  → 결과를 파일로 저장 + 화면 출력

Step 3: kdh-review에 4번째 크리틱 추가 (~1시간)
  → codex 크리틱 에이전트 스폰
  → D1-D6 템플릿 동일 사용
  → 보안/어드버셜 가중치 적용

Step 4: kdh-code-review Phase 4를 MoA로 (~2시간)
  → Hermes MoA 도구 패턴 참조
  → Reference 모델 3개 + Aggregator Opus
  → 3개 크리틱 각각 MoA 적용

Step 5 (선택): 백그라운드 GPT 위임 (~1시간)
  → kdh-build Phase 2에서 버그 시 GPT 위임
  → 결과 폴링 + 반영
```

```
2. GPT 결과는 참고용 — 최종 판정은 항상 Opus (Aggregator)
```
